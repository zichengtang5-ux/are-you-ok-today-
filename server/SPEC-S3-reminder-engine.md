# Plan: Enable后端自动检测未回复用户并推送关心提醒和告警通知，让独居老人及时被发现关心
**Complexity:** Medium — 新模块（device + push）+ 定时任务引擎，跨模块集成（reminder/reply/alert/sms），接口需稳定后再实现。

## Goal
Enable 后端系统 to 自动检测每日提醒窗口内未回复的用户，通过 APNs 推送关心式提醒，宽限期后触发告警并 SMS 通知紧急联系人，使独居用户能被及时发现和关心。

## Deliverables

- Prisma model: `Device` — userId + token + platform，用于存储 APNs device token → 提供推送目标
- API endpoint: `POST /api/device/register` — 接受 token + platform，upsert 到 Device 表 → 前端上报 APNs token 的入口
- Service: `PushService` — APNs 推送服务（mock 模式 + stub），与 SmsService 风格一致 → 关心式提醒和告警推送的发送通道
- Service: `ReminderCronService` — 基于 @nestjs/schedule 的定时任务，每日扫描提醒窗口 → 核心引擎，驱动整个 S3 流程
- Module: `DeviceModule` — 注册 device controller + service → NestJS 模块注册
- Module: `PushModule` — 注册 push service → 全局可用
- Package: `@nestjs/schedule` 依赖安装 → 定时任务基础设施
- 单元测试: device.register.spec.ts + reminder-cron.spec.ts → 覆盖 AC

**Out of scope:**
- APNs 真实证书配置和生产推送（mock 模式 + stub 即可）
- S4 超时告警的完整 SMS 升级逻辑（S3 只触发告警状态变更，S4 细化 SMS 轮次）
- 子女端推送（S5 范围）
- 前端代码

## Acceptance Criteria

| ID | 类型 | 条件 |
|----|------|------|
| AC-1 | Happy path | Given 已登录用户 When POST /api/device/register with valid token+platform Then 返回 `{ message: "设备已注册" }` 且 Device 记录已创建/更新 |
| AC-2 | Happy path | Given 用户未回复且已过提醒窗口 When 定时任务执行 Then GuardStatus 变为 `grace`，APNs 关心式提醒已发送（mock log），30 分钟后再次检测 |
| AC-3 | Happy path | Given 用户处于 grace 状态且宽限期已过仍未回复 When 定时任务再次检测 Then GuardStatus 变为 `alert`，AlertEvent 创建，SMS 通知所有已验证联系人 |
| AC-4 | Error path | Given 无效 token（空字符串） When POST /api/device/register Then 返回 400 错误 |
| AC-5 | Edge case | Given 用户已回复 When 定时任务执行 Then 跳过该用户，不发送任何推送 |
| AC-6 | Edge case | Given 用户处于 paused 状态 When 定时任务执行 Then 跳过该用户 |

## Implementation Path

1. **安装依赖** — `npm i @nestjs/schedule`，为定时任务提供基础设施
2. **Prisma schema 扩展** — 新增 Device model（userId, token, platform, createdAt, updatedAt），User 添加 devices 关系；运行 migration
3. **PushModule + PushService** — mock 模式推送服务，方法：`sendCareReminder(userId)`, `sendAlertNotification(userId, contacts)`；与 SmsService 风格一致（provider=mock|apns）
4. **DeviceModule** — DeviceController（POST /api/device/register）+ DeviceService（upsert token）+ DTO + 单元测试
5. **ReminderCronService** — @nestjs/schedule CronJob，核心逻辑：
   - 查询所有 ReminderConfig（非 paused 用户）
   - 判断当前时间是否在 endTime 之后
   - 检查 DailyRecord 是否已 replied
   - 未回复 → GuardStatus 改为 grace → 调 PushService.sendCareReminder
   - grace 用户 → 检查宽限期是否已过 → GuardStatus 改为 alert → 创建 AlertEvent → SMS 通知联系人
6. **注册 ScheduleModule** — app.module.ts 添加 ScheduleModule.forRoot()
7. **单元测试** — device register + reminder cron（mock PrismaService + PushService + SmsService）
8. **验证** — 编译通过 + 测试通过 + Swagger 文档可见新端点

## Assumptions
- APNs 推送使用 mock 模式开发，真实证书配置不在 S3 范围
- 定时任务使用 @nestjs/schedule 的 CronJob（每分钟检查），不使用 Bull/Redis（当前规模不需要）
- 时区处理沿用 reply.service.ts 的 `todayString()` 模式（UTC+8 hardcode）
- 用户 onboardingStep 为 'completed' 且 isOnboarded=true 才纳入定时任务扫描
