# Plan: Fix 4 backend bugs and align API contracts for frontend S5-S7 integration
**Complexity:** Large — spans 7+ files across 5 modules, adds a new endpoint, modifies Prisma schema, and touches the core reply/pause/alert state machine that other services depend on.

## Goal
Enable the frontend to correctly implement S5-S7 pages (alert detail, pause/resume, account deletion, proxy-reply) by fixing backend state-machine bugs, hardening API contracts, and adding the missing `GET /alert/active` endpoint.

## Deliverables

- **Bug fix**: `GET /reply/status` auto-expires paused state when `PauseLog.endTime <= now` → frontend no longer sees stale `'paused'` after pause window elapses (BUG-1)
- **Bug fix**: `DELETE /reply/today` re-activates resolved `AlertEvent` when undo happens during grace/alert window → state machine stays consistent after undo (BUG-2)
- **Bug fix**: `POST /device/register` deletes all old device tokens before upsert → only latest device receives pushes (BUG-3)
- **Bug fix**: `DELETE /user/account` deletes `VerificationCode` by user's actual phone instead of userId (BUG-4)
- **Contract change**: `POST /reply/today` returns `409 Conflict` when `GuardStatus.status === 'paused'` → frontend can show "请先恢复守护" (B2)
- **New endpoint**: `GET /api/alert/active` returns active alert detail with timeline and contactsNotified (user perspective, no smsRounds) → frontend alert page has data source (B3)
- **Contract change**: `DELETE /user/account` marks other users' `EmergencyContact` rows referencing deleted user's phone with `isAccountDeleted: true` (B5)
- **Contract change**: `POST /guardian/wards/:id/proxy-reply` validates `isBound === true` before allowing proxy reply (B6)
- **Contract change**: `DELETE /reply/today` determines post-undo status based on current time vs reminder window end → returns `'waiting'` if before window end, `'grace'` if after (N1)
- **Schema change**: `EmergencyContact` model gains `isAccountDeleted Boolean @default(false)` field + Prisma migration (B5)
- **Tests**: Updated `*.spec.ts` files covering all changed behavior paths

**Out of scope:**
- APNs actual push delivery (mock mode remains)
- SMS/voice call integration (mock mode remains)
- WebSocket/SSE push for guardian dashboard (Phase 2)
- Any frontend UI changes
- D-class design decisions (require product input)

## Implementation Path

2. **BUG-1: Pause auto-expire in reply status** — modify `ReplyService.getStatus()` to check and auto-expire pause → verify with AC-1

## Open Questions

1. **BUG-2 scope**: When undoing a reply that resolved an alert, should we create a brand new `AlertEvent` or re-open the old one? **Decision: create a new one** — simpler, preserves audit trail, the old resolved alert stays as history.
2. **B5 phone masking**: Should the `isAccountDeleted` contact's phone be masked immediately in the database, or only masked in API responses? **Decision: keep phone as-is in DB, add `isAccountDeleted` flag, mask in API response** — preserves data for potential account restoration scenarios.
3. **N1 undo + alert re-creation timing**: If user undoes reply 2 hours after the window ended (alert was triggered, then resolved by reply), should the new alert start from scratch or inherit the old timeline? **Decision: start fresh** — new AlertEvent with new timeline starting from undo time.
