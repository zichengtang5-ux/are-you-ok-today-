## 二、Phase 1 MVP 需求清单

### REQ-001：用户注册与身份绑定

| 字段 | 内容 |
|------|------|
| **问题** | 独居者需要快速创建守护账户，绑定手机号身份 |
| **目标用户** | 独居者本人 |
| **场景** | 用户首次打开 App，需要快速完成注册 |
| **优先级** | P0（必须） |
| **类别** | 基础功能 |
| **用户故事** | 作为独居者，我希望用手机号验证码快速登录并填写基本信息，以便快速开始使用守护服务 |

**功能点**：
- 手机号 + 短信验证码登录
- 补充填写：昵称、当前住址（用于紧急时刻联系人和 120 快速定位）
- 基本信息页标题强调安全/急救用途（如"你的安全信息"），而非纯身份展示
- 首次使用引导流程：登录+协议 → 设置紧急联系人 → 设置提醒时间 → 授权通知 → 完成
- 引导用户授权 iOS 通知权限（每日提醒必须）
- 登录页引导语突出产品价值："每天一句平安，守护独居的你"

**验收标准**：
- [ ] 手机号验证登录后 4 步内完成注册流程
- [ ] 基本信息页包含当前住址字段，且有明确的用途说明（紧急定位）
- [ ] 未授权通知权限时，明确提示"不授权将无法收到每日平安提醒"
- [ ] 注册完成后自动进入守护首页

### REQ-003：每日"今天还好"提醒

| 字段 | 内容 |
|------|------|
| **问题** | 独居者需要每天确认安全状态，但"签到"太机械，需要温暖的交互体验 |
| **目标用户** | 独居者本人 |
| **场景** | 用户设定每天某个时间段收到温暖的提醒，快捷回复"今天还好" |
| **优先级** | P0（必须） |
| **类别** | 核心功能 |
| **用户故事** | 作为独居者，我希望每天收到一条像朋友问候一样的提醒，点一下就能告诉关心我的人我没事 |

**功能点**：
- 用户设定每日提醒时间窗口（如"每天晚上 8-10 点"），默认窗口 2 小时
- 超时从窗口结束时算起（如窗口 8-10 点，则 10 点开始计算超时）
- 到达提醒时间后，推送 iOS 本地通知："今天还好吗？"
- **通知内快捷回复（Notification Actions）**：用户在通知横幅下拉直接点"今天还好"，1 次点击完成，无需打开 App
- App 内备用入口：一键"今天还好 ✓"按钮
- **温暖反馈**：回复后显示"收到，安心了 ☺" + 显示"已连续 X 天平安"
- 下次提醒时间预告："明天晚上 8 点再见"
- 全局措辞规范：不出现"签到""打卡"等工具化用语

**验收标准**：
- [ ] 在用户设定的时间窗口内准时推送（误差 ±15 分钟）
- [ ] 支持推送内快捷回复，操作 ≤ 1 次点击
- [ ] 回复后有温暖视觉反馈，无冷冰冰的系统提示
- [ ] 用户当天已回复后，不再重复推送
- [ ] 全流程不出现"签到""打卡"字样

### REQ-004：超时检测与联系人通知

| 字段 | 内容 |
|------|------|
| **问题** | 用户未按时回复"今天还好"时，需要通知紧急联系人 |
| **目标用户** | 独居者（被守护）、紧急联系人（被通知） |
| **场景** | 提醒时间窗口结束后用户仍未回复 → 触发通知 |
| **优先级** | P0（必须） |
| **类别** | 核心功能 |
| **用户故事** | 作为独居者，我希望如果我忘了回复或出了事，我的紧急联系人能收到通知来确认我的安全 |

**功能点**：
- 云端定时器：提醒时间窗口截止后，检查是否已回复
- 未回复 → 先给用户发一条**关心式提醒**（再给 30 分钟宽限）：
  - "没收到你的回应，有点担心，看到消息回一下？"
- 宽限后仍未回复 → 状态变为**"告警"**，通知紧急联系人：
  - **免费用户**：SMS 短信通知联系人
  - **付费用户**：SMS 短信 + 语音电话双通道
- 通知内容："[用户昵称] 今天没有回复平安，最后回复时间：XXX，请及时联系确认"
- 按联系人优先级顺序通知（先通知优先级最高的）
- 同一告警事件短信最多发 2 轮（首次 + 10分钟后第二轮）
- 连续 3 天超时触发 → 提示用户调整提醒时间，减少无效触发
- **子女端场景**：父母不回复时标记**"未确认"**状态（非告警），通知子女"父母今天还没回复"，子女可代确认"TA没事"
- 状态区分：**"未确认"**（温和提醒阶段）≠ **"告警"**（联系人已通知阶段）

**验收标准**：
- [ ] 提醒截止后 30 分钟宽限期内回复 → 不触发联系人通知
- [ ] 宽限期间的关心式提醒语气温暖，非系统告警风格
- [ ] 免费用户联系人收到 SMS 短信通知
- [ ] 付费用户联系人同时收到 SMS + 语音电话通知
- [ ] 宽限结束后 5 分钟内完成联系人通知发送
- [ ] 通知内容包含用户昵称、最后回复时间、建议行动
- [ ] 联系人收到短信通知后可通过深链接进入 App 查看状态
- [ ] 子女端可代确认"TA没事"，解除未确认状态
- [ ] "未确认"和"告警"状态在 UI 上有明确视觉区分
- [ ] 同一告警事件短信不超过 2 轮

### REQ-005：联系人确认/接管

| 字段 | 内容 |
|------|------|
| **问题** | 联系人收到通知后需要反馈"已处理"，防止误报升级 |
| **目标用户** | 紧急联系人 |
| **场景** | 联系人收到超时通知 → 联系用户确认安全 → 在 App 标记"已处理" |
| **优先级** | P0（必须） |
| **类别** | 核心功能 |
| **用户故事** | 作为紧急联系人，我希望在确认被守护者安全后，能标记"已处理"，让系统知道情况已解决 |

**功能点**：
- 联系人打开 App 后看到当前告警状态
- 两个操作按钮："已联系，TA没事 ✓" / "联系不上，需要帮助"
- 点击"已联系" → 告警解除，记录处理时间
- 点击"需要帮助" → 显示建议行动清单（拨打用户电话、拨打120、联系其他联系人）
- 如所有联系人 10 分钟内均未确认 → 发送第二轮短信提醒给联系人
- **Phase 1 不包含自动拨打120**——"需要帮助"时提供手动操作指引

**验收标准**：
- [ ] 联系人可在通知消息中直接进入确认页面
- [ ] "已联系"操作后，告警状态立即解除
- [ ] "需要帮助"提供清晰的行动建议，包括120拨打指引
- [ ] 10分钟无联系人确认 → 第二轮短信提醒
- [ ] 所有操作有记录日志（谁在什么时间做了什么）

## 四、Phase 1 → Phase 2 衔接

Phase 1 上线后需要验证的关键指标，决定 Phase 2 启动时机：

| 指标 | 目标值 | 意义 |
|------|--------|------|
| DAU / 注册量 | ≥ 30% | 需求真实性 |
| 日回复率 | ≥ 70% | 用户习惯养成 |
| 超时触发率 | ≤ 15% | 提醒时机是否合理 |
| 联系人响应率 | ≥ 80% | 联系人链路有效性 |
| 免费→付费转化率 | ≥ 5% | 商业模式验证 |
| 子女付费占比 | ≥ 20% | 路径D验证 |

Phase 2 启动条件：DAU ≥ 1000 且 日回复率 ≥ 60%

Phase 2 核心新增：
- Apple Watch HealthKit 接入 → 心率异常联动
- 微信小程序辅助入口（联系人看板/获客渠道）
- 分级响应链完整版（电话确认 → 自动拨打120）
- 多渠道通知（电话+短信+语音）
- 高度自定义响应链
- 保险合作数据准备

## 五、开放问题（需求层）— 已全部确认

1. ~~**提醒时间窗口 vs 固定时间点**~~ → **决策：时间窗口**。用户设定2小时提醒窗口（默认值），超时从窗口结束时算起 + 30分钟宽限。灵活适配不同生活节奏，减少"错过即误报"。
2. ~~**联系人通知的短信成本**~~ → **决策：免费/付费分通道**。免费用户 SMS 短信通知联系人（≤0.08元/事件）；付费用户 SMS + 语音电话双通道。同一告警事件短信最多2轮。连续3天超时触发 → 提示用户调整提醒时间。
3. ~~**父母不配合回复怎么办**~~ → **决策：温和引导 + 子女代确认**。父母不回复时标记"未确认"状态（非告警），通知子女"父母今天还没回复"，子女可代确认"TA没事"。区分"未确认"和"告警"两种状态，不产生误报。
4. ~~**多时区/出差场景**~~ → **决策：MVP不处理**。提醒时间按用户设定的本地时间推送，Phase 2 再考虑时区自动调整。
