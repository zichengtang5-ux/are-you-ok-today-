# 独居提醒 — 产品探索文档

> 文档类型：产品探索 & 竞品调研
> 创建日期：2026-06-22
> 状态：历史探索材料；当前产品基线见 `独居提醒-PRD-Phase1-v2.0.md`

---

## 1. 产品概念

**一句话定义**：面向独居人士的安全守护小程序——每日签到确认安全，无响应时自动通知紧急联系人和医疗机构，结合 Apple Watch 健康监测实现主动式异常预警。

### 目标用户
- 独居青年/中年（约 1.2 亿中国独居人口）
- 独居老人及其子女
- 有慢性病/心脏疾病风险的独居者
- 关注自身安全的独居女性

### 核心痛点
| 痛点 | 场景 |
|------|------|
| 出事无人知 | 独居者突发疾病、摔倒、昏迷，无人发现 |
| 心理焦虑 | 担心"如果我出事了，多久才会有人知道" |
| 联系中断 | 手机不在身边/无法操作时缺乏兜底机制 |
| 子女担忧 | 远在他城的父母/子女无法确认安全状态 |

## 2. 竞品分析

### 2.1 直接竞品

| 产品 | 平台 | 核心机制 | 签到周期 | 通知方式 | 特色 | 不足 |
|------|------|----------|----------|----------|------|------|
| **死了么** | Android | 每日签到按钮 | 连续3日未签到触发 | 邮件通知紧急联系人 | 极简轻量、支持优先级联系人 | 仅邮件通知、3天周期太长、无穿戴设备联动 |
| **独居签到** | Android | 一键签到 | 每日 | 未明确 | 零门槛、无需注册 | 功能单一、无智能监测 |
| **独居宝** | Android | 安全守护 | - | - | - | 信息有限 |
| **在么在么** | iOS/Android | 安全确认 | - | - | - | 信息有限 |
| **平安签** | iOS | 健康打卡 | 每日 | - | iOS 原生 | 偏打卡习惯，非紧急守护 |
| **活着么** | Android | 签到守护 | - | - | 类似"死了么" | 同质化严重 |

### 2.2 间接竞品

| 产品/功能 | 类型 | 可借鉴点 |
|-----------|------|----------|
| **Apple Watch 摔倒检测** | 系统级功能 | 自动倒计时 → 拨打急救 → 通知联系人的完整流程 |
| **Apple Watch 心脏健康通知** | 系统级功能 | 高/低心率通知、不规则心律通知、房颤历史 |
| **iPhone 崩溃检测** | 系统级功能 | 严重车祸自动检测与报警 |
| **社区医生签到** | 社区服务 | "没签到就发邮件"模式，30元/月守护独居者 |

### 2.3 竞品痛点总结

1. **通知渠道单一**：大多仅支持邮件，紧急场景下邮件几乎无用
2. **签到周期设置不合理**：3天才触发太慢，但每天又可能打扰
3. **无穿戴设备联动**：没有与 Apple Watch/智能手环的心率、摔倒数据打通
4. **缺乏主动预警**：只能被动等待签到，无法基于健康数据主动报警
5. **体验粗糙**：UI 简陋、功能同质化、缺乏信任感设计

## 3. Apple Watch 紧急呼救与健康监测功能深度分析

### 3.1 摔倒检测（Fall Detection）

**工作原理**：
1. Apple Watch 内置加速度计和陀螺仪持续监测运动状态
2. 检测到**严重摔倒**时：轻点手腕 + 响起警报音
3. 若用户**无回应且约1分钟未动弹** → 启动 **30秒倒计时**
4. 倒计时结束 → **自动拨打紧急服务（120/110）**，共享位置信息和医疗急救卡
5. 通话结束后 → **通过 iMessage 通知紧急联系人**，附带位置信息

**触发条件**：
- 65岁及以上用户：默认开启
- 18-64岁用户：需手动开启（设置 → 摔倒检测）
- 需要蜂窝网络或卫星连接才能拨打紧急电话

**对产品的启示**：
- ✅ 倒计时机制给用户"误触取消"的机会，减少误报
- ✅ 分级通知：先急救 → 再联系人，优先级清晰
- ✅ 位置信息自动附带，减少救援响应时间
- ⚠️ 系统级功能，第三方 App 无法直接调用摔倒检测事件

### 3.2 心脏健康监测

**已有功能**：
| 功能 | 说明 | 触发方式 |
|------|------|----------|
| **高心率通知** | 静息状态下心率超过设定阈值（默认 120bpm） | 后台自动检测 |
| **低心率通知** | 静息状态下心率低于设定阈值（默认 40bpm） | 后台自动检测 |
| **不规则心律通知** | 检测到疑似房颤的不规则心律 | 后台自动检测 |
| **房颤历史** | 追踪房颤发作频率 | 后台持续监测 |
| **心电图（ECG）** | 30秒单导联心电图 | 用户主动触发 |

### 3.3 HealthKit API 与第三方 App 集成

**可用能力**：

| API | 能力 | 对独居提醒的价值 |
|-----|------|------------------|
| `HKHeartbeatSequenceQuery` | 查询心跳序列数据 | 后台监控心率异常模式 |
| `HKObserverQuery` + 后台交付 | 数据变化时唤醒 App | 实现"后台守护"而非前台运行 |
| `HKStatisticsCollectionQuery` | 聚合统计心率数据 | 分析静息心率趋势，发现渐变异常 |
| `HKWorkoutSession` | 运动期间的实时数据 | 运动安全守护 |
| `HKQuantityType.heartRate` | 实时心率读取 | 配合自定义阈值报警 |
| `HealthKit Background Delivery` | App 被杀后仍可接收数据 | **关键能力**：确保守护不中断 |

**限制**：
- ❌ 第三方 App **无法直接接收摔倒检测事件**（系统独享）
- ❌ watchOS 26 上 `HKObserverQuery` 后台交付存在稳定性问题（[开发者论坛反馈](https://forums.developer.apple.com/forums/thread/801627)）
- ❌ HealthKit 数据仅存储在用户设备上，无法直接同步到云端服务器
- ⚠️ 需要用户主动授权心率数据读取权限

**可行集成路径**：
```
Apple Watch → HealthKit 心率数据 → iOS App（后台）→ 
  检测异常 → 推送通知 → 用户确认/超时 → 
  自动通知紧急联系人（短信/电话/微信）
```

## 4. 主动式体验设计探索

### 4.1 "主动式" vs "被动式"的核心差异

| 维度 | 被动式（现有竞品） | 主动式（我们的方向） |
|------|-------------------|---------------------|
| 触发 | 用户主动签到 | 系统主动关怀 + 用户轻量确认 |
| 数据 | 无（仅签到状态） | 心率、运动、位置、手机使用模式 |
| 响应 | 超时 → 通知联系人 | 实时异常 → 分级响应 |
| 体验 | 每天"打卡"有负担 | 无感守护，异常时才打扰 |

### 4.2 主动式守护的分层设计

```
┌─────────────────────────────────────────────┐
│  Layer 1: 无感守护（日常，不打扰）            │
│  - Apple Watch 心率后台监测                   │
│  - 手机使用模式分析（是否有活动）              │
│  - 步数/运动数据（是否有日常活动）            │
├─────────────────────────────────────────────┤
│  Layer 2: 轻量确认（有轻微疑虑时）            │
│  - 推送一条温暖的消息："今天还好吗？点一下 ✓" │
│  - 语音助手确认："小X，你还好吗？"            │
│  - 智能时机选择（根据用户日常作息）            │
├─────────────────────────────────────────────┤
│  Layer 3: 升级确认（较长时间无回应）           │
│  - 多次推送 + 逐渐增强的提醒                  │
│  - 拨打电话确认                              │
│  - 倒计时机制（参考 Apple Watch 摔倒检测）     │
├─────────────────────────────────────────────┤
│  Layer 4: 紧急响应（确认异常或持续无回应）     │
│  - 通知紧急联系人（电话 + 短信 + 位置）        │
│  - 联系社区/物业（如有对接）                  │
│  - 拨打120（最后兜底）                       │
│  - 共享实时位置                              │
└─────────────────────────────────────────────┘
```

### 4.3 关键体验设计点

#### 签到体验
- **不要叫"签到"**：用"报平安"、"今天还好"、"我没事 ✓" 等温暖措辞
- **一键操作**：推送通知直接内嵌确认按钮，无需打开 App
- **智能时机**：学习用户日常习惯，在自然间隙提醒（如起床后、午餐时）
- **渐进式频率**：新用户每日确认 → 一段时间后用户可选"每2天"→ 信任度建立后"每周"

#### 紧急联系人体验
- **多通道通知**：短信 + 电话 + 微信模板消息（不依赖单一渠道）
- **通知内容**：包含位置、最后活动时间、心率数据（如有）、建议行动
- **联系人看板**：联系人可通过小程序查看被守护者的最近安全状态
- **双向绑定**：子女可主动查看父母状态，无需等触发

#### 信任感设计
- **透明机制**：用户清楚知道"谁会收到通知"、"什么情况下会通知"
- **误报快速取消**：倒计时期间一键取消，避免"狼来了"效应
- **隐私承诺**：健康数据仅在本地处理，不上云
- **可调节灵敏度**：用户自定义"多久没回应算异常"

## 5. Apple Watch 联动方案设计

### 5.1 方案一：HealthKit 深度集成（推荐）

```
┌──────────────┐    HealthKit     ┌──────────────┐
│  Apple Watch │ ───────────────→ │   iOS App    │
│  (心率/运动)  │                  │  (后台守护)   │
└──────────────┘                  └──────┬───────┘
                                         │
                              ┌──────────┼──────────┐
                              ▼          ▼          ▼
                        心率异常检测  活动模式分析  签到提醒
                              │          │          │
                              └──────────┼──────────┘
                                         ▼
                                  分级响应引擎
                              ┌──────────┼──────────┐
                              ▼          ▼          ▼
                          轻量确认    升级确认    紧急通知
```

**优势**：
- 利用 Apple Watch 已有的传感器，无需额外硬件
- HealthKit 后台交付保证 App 被杀后仍能工作
- 心率数据可自定义阈值，适配不同用户健康状况

**挑战**：
- 仅覆盖 Apple Watch 用户（市场份额有限）
- HealthKit 后台交付在 watchOS 26 上有稳定性问题
- 需要 iOS App 配合（小程序无法直接访问 HealthKit）

### 5.2 方案二：小程序 + iOS App 混合架构

```
┌─────────────────┐           ┌─────────────────┐
│  微信小程序       │ ◄──────► │   iOS 原生 App   │
│  (轻量签到/通知)  │  WebSocket│  (HealthKit 桥接) │
└────────┬────────┘           └────────┬────────┘
         │                             │
         └─────────┬───────────────────┘
                   ▼
            ┌──────────────┐
            │   云端服务     │
            │  (守护引擎)    │
            └──────┬───────┘
                   │
          ┌────────┼────────┐
          ▼        ▼        ▼
       紧急联系人  120急救   社区/物业
```

**架构要点**：
- **小程序**：签到入口、紧急联系人管理、状态查看（轻量化、易传播）
- **iOS App**：HealthKit 数据桥接、后台守护引擎、Apple Watch 联动
- **云端服务**：守护状态机、超时计算、多渠道通知调度

### 5.3 方案三：纯小程序 + 手机传感器（MVP 快速验证）

```
┌──────────────────────────┐
│       微信小程序           │
│  - 每日提醒签到           │
│  - 微信步数（间接活动数据）│
│  - 小程序订阅消息         │
│  - 微信服务通知           │
└──────────┬───────────────┘
           ▼
    ┌──────────────┐
    │   云端服务     │
    │  超时检测引擎  │
    └──────┬───────┘
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
  短信    电话    微信模板消息
```

**优势**：开发成本低、用户无需下载 App、微信生态通知能力强
**限制**：无心率数据、无法后台持续运行、依赖用户主动打开小程序

## 6. 建议的产品路线

### Phase 1：MVP 验证（1-2个月）
- **方案三**（纯小程序）快速上线
- 核心功能：每日签到提醒 + 超时通知紧急联系人
- 验证需求真实性和用户留存

### Phase 2：增强守护（3-4个月）
- 引入 **iOS App**（方案二），接入 HealthKit
- Apple Watch 心率异常监测
- 多渠道通知（短信 + 电话 + 微信）
- 智能签到时机学习

### Phase 3：完整生态（5-6个月）
- 主动式分层守护引擎上线
- 紧急联系人看板
- 社区/物业对接
- Android 智能手环适配（扩大覆盖面）
- 子女端"关怀模式"

## 7. 关键风险与开放问题

### 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 误报率高 → 用户关闭通知 | 产品失效 | 渐进式灵敏度 + 误报快速取消 |
| 漏报 → 安全事故 | 信任崩塌 + 法律责任 | 明确免责条款 + 多重兜底 |
| HealthKit 后台不稳定 | Apple Watch 联动失效 | 本地 + 云端双重检测 |
| 隐私合规 | 下架/处罚 | 健康数据本地化处理、隐私政策 |
| 小程序后台限制 | 微信无法后台持续运行 | iOS App 配合、订阅消息兜底 |
| Apple Watch 用户占比低 | 覆盖不足 | Phase 3 扩展到 Android 手环 |

### 开放问题

1. **商业模式**：免费+增值？付费订阅？保险合作？政府/社区采购？
2. **法律责任边界**：如果系统未能及时通知导致事故，产品是否承担责任？
3. **医疗机构对接**：自动联系医院的可行性（120 vs 指定医院 vs 社区医生）？
4. **Apple Watch vs 小米手环**：是否初期只支持 Apple Watch，还是同步适配？
5. **小程序 vs App 优先**：微信小程序传播快但能力弱，App 能力强但获客难——如何平衡？
6. **用户隐私预期**：心率数据、位置数据的使用边界在哪里？
